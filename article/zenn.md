# 構造化された議論サービスを Go + Lit で実装した

議論をしていると、途中で「今どの前提について話しているのか」「この発言は賛成なのか反対なのか」が分からなくなることがあります。

時系列に流れるチャットは気軽ですが、論点が増えるほど、会話の構造を頭の中だけで追うのが難しくなります。そこで作っているのが、コメントを木構造で管理し、さらにコメントの役割まで扱える議論プラットフォーム Supportocol です。

この記事では、Supportocol をどんな構成で実装しているのかを、リポジトリの構造と採用ライブラリをもとにまとめます。

アプリの概要についてはこちらの紹介記事をご覧ください。

## 何を作っているのか

Supportocol には大きく 2 つの文脈があります。

- 学習コンテキスト: 1 人で考えを整理する場
- 対話コンテキスト: 他者と構造化された議論をする場

学習コンテキストでは、既存のコメントからコメントフレームを動的に導出できます。たとえば、ある議論の中で「質問」に対して「回答」が付き、「提案」に対して「賛成」「反対」「改善案」が付いているなら、その関係をあとから取り出して対話コンテキストに持ち込めます。

対話コンテキストでは、そのコメントフレームを固定して、次にどんな種類のコメントを返せるかを制約します。これによって、議論が自由すぎて拡散するのを防ぎつつ、論点の流れを追いやすくしています。

さらに、コメントとは別レイヤーで「指摘」を付けられるようにしています。これは「論理の飛躍がある」「前提と矛盾している」といったメタな問題提起を、議論本体の木構造を壊さずに扱うためです。

## 全体構成

リポジトリは大きく分けると、Go の API / ドメイン層、Lit 製のフロントエンド、AWS CDK によるインフラで構成しています。

```text
.
├── internal/
│   ├── identity/
│   ├── workspace/
│   ├── learning/
│   └── dialogue/
├── view/
└── cdk/
```

設計の軸は、議論サービスの関心をそのまま境界づけられたコンテキストに分けることでした。

- identity: 認証とユーザー
- workspace: ワークスペース、プロジェクト、メンバー管理
- learning: 個人の思考整理
- dialogue: 他者との構造化議論

Go 側の各コンテキストは、ほぼ共通して次のようなレイヤ構成にしています。

- api: HTTP ハンドラーと OpenAPI 生成コード
- usecase: アプリケーションロジック
- domain: エンティティ、値オブジェクト、ドメインサービス
- infra: DB や外部サービスとの接続
- container.go: DI の組み立て

議論というドメインは「UI の都合」と「ルールそのもの」が混ざると急に壊れやすくなるので、少なくとも usecase と domain は分けておいたほうが後から楽でした。

## バックエンドは Go 1.26 + OpenAPI 生成 + Lambda

バックエンドは Go 1.26 で実装しています。API のエントリポイントでは、各コンテキストのコンテナを組み立てて HTTP ハンドラーに束ねています。

採用している主なライブラリは次のとおりです。

- ogen: OpenAPI から Go のサーバーコードを生成
- go-jet: MySQL 向けのクエリコード生成
- golang-migrate: マイグレーション管理
- aws-sdk-go-v2: AWS 連携
- awslabs/aws-lambda-go-api-proxy: Lambda 上での HTTP 実行
- google.golang.org/genai: Gemini を使ったコメント生成

OpenAPI を契約の中心に置いているのが、この構成のかなり重要なポイントです。

各コンテキストの api パッケージでは、OpenAPI 定義から ogen でサーバー側コードを生成しています。手書きしているのは、生成されたインターフェースを満たすハンドラーと、usecase への変換処理です。

これによって、API 仕様のズレが「なんとなくフロントとバックで合わない」という形ではなく、生成コードの差分として見えるようになります。

## フロントエンドは React ではなく Lit を使った

フロントエンドは Lit 3 を使っています。個人的に、Supportocol の UI は「小さい Web Components の組み合わせ」と相性がよいと感じたためです。

採用している主なライブラリは次のとおりです。

- lit
- @lit-labs/router
- @lit/context
- @lit/task
- @lit/localize
- openapi-fetch
- vite
- vite-plugin-pwa
- Storybook
- Vitest
- Playwright

ルーティングは @lit-labs/router、状態の受け渡しは @lit/context、非同期処理は @lit/task を使っています。
たとえばアプリのルートでは、ユーザー情報を取得して Context に流し、learning / dialogue / workspace / identity の各ルートに分岐させています。

機能ごとの分割は view/src/feature 配下に寄せています。

- feature/learning
- feature/dialogue
- feature/workspace
- feature/identity
- feature/marketing

さらに各 feature の中を、component と ui に分けています。

- component: API 呼び出しや状態管理を担当する Container
- ui: 表示に専念する Presenter

議論 UI はどうしても複雑になりやすいので、この分離はかなり効きました。特にコメントツリーまわりは、見た目の責務と状態変化の責務を混ぜるとすぐ読めなくなります。

## OpenAPI をフロントにも流している

Supportocol では、OpenAPI をバックエンドだけで閉じず、フロントの型生成にも使っています。

view/package.json には次のようなスクリプトがあります。

- identity, learning, dialogue, workspace の各 OpenAPI 定義から openapi-typescript で型生成
- 生成した型を openapi-fetch でそのまま API クライアントに使う

これで、サーバーのレスポンス型とフロントの期待値がズレにくくなります。

議論サービスでは、コメント、コメントフレーム、指摘、権限など、UI が参照する構造が細かくなりがちです。ここを hand-written な型で持つと、いずれ更新漏れが出るので、契約から生成する運用に寄せました。

## Supportocol ならではの実装 1: コメントフレームを議論から導出する

一番 Supportocol らしいのはここです。

学習コンテキストでは、既存のコメント列からコメントフレームを導出しています。コメントの種類だけを集めるのではなく、親コメントの種類と子コメントの種類の関係も拾って、

- どんな種類が存在するか
- どの親にどの子を付けられるか

をフレームとして取り出します。

この仕組みによって、最初から厳密な議論ルールを押し付けるのではなく、まずは自分の思考過程からルールを発見し、その後の対話で再利用できます。

「思考整理の場」と「他者との議論の場」で必要な制約は違うので、ここはかなり大事な設計ポイントでした。

## Supportocol ならではの実装 2: 指摘をコメントツリーから分離した

議論の中では、「それは論点がずれている」「根拠が弱い」といったメタな評価が必要になります。

ただし、それを通常コメントとしてぶら下げると、議論本体の木構造とメタ議論が混ざります。すると、どこまでが本論で、どこからがレビューなのかが見えにくくなります。

そこで Supportocol では、コメントに対して Issue を付ける形にしました。これで、

- 議論の流れ
- 発言への批判や検証

を別レイヤーとして扱えます。

普通のスレッドだとこの 2 つが混ざりやすいので、構造を保つうえでかなり効いています。

## Supportocol ならではの実装 3: AI でコメント案を生成する

学習コンテキストでは、Gemini を使ってコメント案を生成する機能を入れています。

実装としては、いきなり UI からモデルを叩くのではなく、SQS に積んで非同期で処理する構成です。

生成時には、

- プロジェクト前提
- 議論前提
- 祖先コメントの経路
- すでに存在する同種コメント

を集めてプロンプトを構築し、同じ内容を重複させず、前提に矛盾しないコメント案を返すようにしています。

ここで重要だったのは、AI に自由回答させるのではなく、JSON スキーマ付きでレスポンスを制約することでした。生成結果をそのまま proposed 状態のコメントとして扱えるので、UI 側での取り回しがかなり安定します。

## 認証、設定、配備まわり

認証は Google ログインを採用しています。フロントで Google Identity Services を使い、バックエンドの identity コンテキストで受けて JWT を発行する流れです。

設定値は AWS Systems Manager Parameter Store から読み出しています。バックエンドだけでなく、view 用の環境変数ファイルも Go の小さなコマンドで生成しています。

デプロイは AWS CDK を使っています。バックエンドは Lambda、フロントは Vite ビルドした成果物を配る構成です。PWA も有効にしていて、vite-plugin-pwa で manifest と service worker を生成しています。

## なぜこの構成にしてよかったか

実装していて特によかったのは、次の 3 点です。

1. ドメインの境界を先に切ったこと
2. OpenAPI を中心にしてサーバーとフロントをつないだこと
3. フロントを feature 単位で分割し、さらに Container / Presenter を分けたこと

learning と dialogue を別コンテキストとして分けたのは正解でした。見た目は似ていても、自由に構造を育てる場と、ルールのある議論の場では、操作の意味がかなり違うからです。

## おわりに

Supportocol は、ただコメントを投稿するサービスではなく、「発言の役割」と「前提の構造」を扱うためのアプリです。

そのぶん実装でも、一般的な掲示板やチャットより一段だけドメインの形が複雑になります。だからこそ、Go のコンテキスト分割、OpenAPI 起点の型共有、Lit による機能分割がうまく噛み合いました。
